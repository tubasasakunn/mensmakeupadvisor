# ADR を起こす

引数（決定のタイトル）: $ARGUMENTS

## 手順

1. `docs/adr/` を Glob で一覧し、既存の最大番号 + 1 を新しい番号にする（4 桁ゼロ埋め）。
2. `docs/adr/template.md` を読み、構成に従って `docs/adr/NNNN-短いケバブケース.md` を書く。
   - タイトルは「決定を一文で」。$ARGUMENTS が空なら直近の会話から決定内容を要約する。
   - **Context には制約を、Alternatives には却下した案と理由を必ず書く**。
     ここが「コードから読み取れない情報」であり ADR の存在価値。
   - プロダクトの判断フィルター（あれば）に関わる決定なら必ず触れる。
   - **再考の条件**を具体的に書く（「SDK が○○に対応したら」「Xcode 27 が出たら」等）。
3. `docs/adr/README.md` の一覧表に行を追加する。
4. 既存 ADR を覆す決定なら、古い ADR の Status を `Superseded by NNNN` に更新する
   （本文は書き換えない）。
5. 決定がコーディング規約に影響するなら `.claude/rules/` も更新する。
   罠・手順に影響するなら `CLAUDE.md` も更新する。

## ADR を書くべき判断の目安

- 「なんでこうなってるの？」と半年後に聞かれそうな決定
- ベストプラクティスから**あえて**外れる決定
- 新しい依存ライブラリ・新しいターゲット・データスキーマの変更
- 一度やめた方式に戻らないための記録
